# 计算机科学知识萃取项目

## 项目概述

本仓库用于从计算机科学各子领域的英文技术文章、博客和文档中提炼核心知识点，输出高可读性的结构化中文 Markdown 摘要文件。输入为 URL，输出为 `articles/[category]/` 目录下的 `[slug].md` 文件，并可通过 `npm run serve` 在浏览器中浏览。

`articles/` 目录下按子领域（category）分类组织，每个子领域对应一个子目录（如 `vibe-coding/`、`databases/`、`os/` 等）。侧边栏和标签索引由 `gen-sidebar.js` 自动扫描 `articles/` 下所有 `.md` 文件的前置元数据（Frontmatter）生成。

## 工作流

输入英文技术文章 URL 后，按以下流程处理：

1. **获取内容**：使用 WebFetch 工具抓取 URL（优先请求 Markdown 格式），获取完整的文章正文。
2. **信息萃取**：严格遵循下文的"内容萃取规范"和"呈现与表达规范"，将原文提炼为结构化中文摘要。
3. **文件命名**：从原文标题中提取关键词，生成英文 slug 短名（全小写，空格与标点替换为连字符 `-`，去除停用词如 a/an/the/of/for/is）。示例：原文 "Context Engineering Best Practices for AI-Driven Dev Teams (2026)" → 文件名 `context-engineering-best-practices.md`。
4. **保存输出**：将完整的 Markdown 摘要写入 `articles/[category]/` 目录。`category` 根据文章主题确定——优先使用已有分类目录名；若为全新子领域，在 `articles/` 下新建对应目录。文件名如上一步确定。
5. **补充元数据**：在新生成的 `.md` 文件顶部补充 YAML Frontmatter，格式如下：

```yaml
---
title: 文章中文标题
category: 分类目录名      # 对应 articles/ 下的子目录名称，如 databases、os、vibe-coding
tags: [标签1, 标签2]     # 根据文章内容确定 2-4 个标签
date: YYYY-MM-DD
---
```

6. **刷新索引**：运行 `npm run gen` 重新生成侧边栏及标签索引页。

> 若用户明确表示只需在对话中浏览摘要而无需保存文件，则跳过步骤 3-6，直接在对话中输出完整 Markdown。

## 核心准则

1. **信息密度优先**：去除口语化废话、重复性铺垫、过渡性寒暄，只保留有价值的观点、事实、数据、步骤与洞察。让读者无需阅读原文，也能掌握核心干货与关键洞察——而非复述所有细节。
2. **严禁幻觉**：严格基于原文进行提炼。若原文中未涉及某些板块，则直接略过，**绝对不要为了凑齐所有板块而编造原文中不存在的信息**。
3. **思维链先行**：在正式输出总结前，请先在内心通读并拆解文章逻辑，确保不遗漏任何关键论点，然后再按照以下要求输出结构化总结。
4. **超长截断策略**：若原文超出上下文窗口导致无法一次性完整萃取，优先保证"核心论点"与"方法论"的完整性，并在文末明确标注"因篇幅限制，部分次要技术细节已省略"。截断场景下仍须按工作流生成并保存为 `.md` 文件。

## 内容萃取规范

以下类别若原文有所涉及，则必须覆盖；若确实没有，则直接略过，不强行占位：

- **核心论点**：文章最想传达的 1-3 个中心思想
- **前置知识背景**：理解原文所需的基础知识（若有，如特定工具、框架或理论前提的介绍）
- **关键概念与定义**：新名词、新方法论、专有缩写及语境解释
- **方法论 / 最佳实践**：可操作的步骤、流程、原则
- **技术细节与常见陷阱**：代码逻辑、架构决策、坑点、注意事项

## 呈现与表达规范

### 隐喻与行业术语还原

- 若原文使用非技术词汇做隐喻（如 Constitution、Silver bullet、Greenfield project），必须转换为国内技术圈通行的标准表达（如"项目核心准则"、"银弹/万能药"、"从零新建项目"），不得生硬直译。
- 如果某些隐喻没有公认的标准中文表达，则保留原文比喻，但紧跟括号补充准确的技术含义解释。
- 多义词（release、handle、consume、context 等）根据具体子领域的技术场景选择最准确的中文表达。

### 术语与专有名词规范

- 专业术语对齐国内主流技术社区及官方中文文档的通用译法，保持平实、准确，禁止自造词。
- 缩写首次出现时，除业界极通用缩写（如 API、HTTP、JSON、SQL）外，均需补充中文全称或一句话解释。宁可多注，不可让读者产生理解障碍。
- 框架、工具、库、API 名称保留英文原样及官方大小写（如 Cursor、Replit、Lovable、React、Supabase、Kubernetes）。

> **子领域术语纠偏示例（AI / LLM 语境）**：Agent → 智能体（非"代理"），Prompt → 提示词，Hallucination → 幻觉，Context window → 上下文窗口，Context decay → 上下文衰减/丢失，Agentic engineering → 智能体工程，Vibe Coding → 氛围编程。其他子领域（如数据库、网络、安全等）同样需遵循该领域国内主流译法，不可自造词。

- 对于文章中首次出现的**关键专业术语、方法论、框架别名**，均采用 **中文（English）** 格式标注一次，后续直接使用中文或英文原名。示例：依赖注入（Dependency Injection）、思维链（Chain-of-Thought）、共识算法（Consensus Algorithm）。

### 代码与原文保留

- 所有原文中出现的**代码示例、命令行、配置文件片段、API 名称、文件路径、ASCII 目录树结构**必须原样保留，不做任何省略、翻译或改动。
- 若原文有对比性内容（如声明式 vs 命令式，有锁 vs 无锁并发），优先使用 Markdown 表格结构化呈现，使差异一目了然。

## 输出格式

- 全程使用 Markdown 排版。
- 如果原文有明确的标题，直接沿用作为 H1 标题；若无标题，则根据文章最核心的主题，提炼一个简洁且包含关键词的中文标题作为 H1。
- 使用 H2/H3 标题划分知识板块（如"核心论点"、"关键概念"、"方法论与最佳实践"、"技术细节与陷阱"等），辅以列表、加粗、引用块（`>`）突出信息层级。
- 结尾必须附上一句 **"一句话总结"**（≤40 字，提炼全文最核心的一个观点）。如果文章内容有明显的受众区分（如"适合 Vibe Coding 新手""适合资深全栈开发者"），可再增加一句 **"对你有何价值"**，直接点明适用场景或可解决的实际问题。
