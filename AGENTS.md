# 计算机科学知识萃取项目

## 项目概述

本仓库用于从计算机科学各子领域的英文技术文章、博客和文档中提炼核心知识点，输出高可读性的结构化中文 Markdown 摘要文件。输入为 URL，输出为 `articles/[category]/` 目录下的 `[slug].md` 文件，并可通过 `npm run serve` 在浏览器中浏览。

`articles/` 目录下按子领域（category）分类组织，每个子领域对应一个子目录（如 `vibe-coding/`、`databases/`、`os/` 等）。侧边栏和标签索引由 `scripts/gen-sidebar.js` 自动扫描 `articles/` 下所有 `.md` 文件的前置元数据（Frontmatter）生成。

## 工作流

输入英文技术文章 URL 后，按以下流程处理：

1. **获取内容**：使用 WebFetch 工具抓取 URL（优先请求 Markdown 格式），获取完整的文章正文。

2. **思维链拆解**：在输出摘要前，按以下子步骤拆解文章：
   - 扫描并列出原文的 H1/H2/H3 标题大纲；
   - 逐段标注核心观点、数据、步骤、代码等关键信息点；
   - 确认"萃取与表达总则"中哪些板块原文有涉及、哪些可略过。

3. **信息萃取**：严格遵循下文的"萃取与表达总则"，将原文提炼为结构化中文摘要。

4. **文件命名**：从原文标题中提取关键词，生成英文 slug 短名（全小写，空格与标点替换为连字符 `-`，去除停用词如 a/an/the/of/for/is，保留数字）。示例：原文 "Context Engineering Best Practices for AI-Driven Dev Teams (2026)" → 文件名 `context-engineering-best-practices.md`。

5. **确定分类目录**：`category` 根据文章主题确定 —— 优先使用 `articles/` 下已有分类目录名；若为全新子领域，在 `articles/` 下新建以全小写连字符命名的目录（如 `distributed-systems/`）。若文章涉及多个领域，以最核心主题为准。

6. **保存输出**：将完整的 Markdown 摘要写入 `articles/[category]/` 目录。文件名如步骤 4 确定。
   - **H1 标题须包含原文链接**：文件正文的 H1 标题格式为 `# [总结后的中文标题](原文URL)`，确保读者可点击标题直达原文。

7. **补充元数据**：在新生成的 `.md` 文件顶部补充 YAML Frontmatter，格式如下（`---` 为必须的分隔符）：

```yaml
---
title: 文章中文标题
category: 分类目录名          # 对应 articles/ 下的子目录名称
tags: [标签1, 标签2]          # 根据文章内容确定 2-4 个标签
date: YYYY-MM-DD              # 原文发表日期；若不可查，填生成日期
order: 数字                    # 该分类下的排序权重，值越小越靠前（可选）
---
```

8. **刷新索引**：运行 `npm run gen` 重新生成侧边栏及标签索引页。

### 异常处理

- **抓取失败**：若 WebFetch 无法获取内容（超时、403、paywall），向用户报告原因并询问是否提供替代 URL 或手动粘贴原文。
- **标题缺失**：若原文无明确标题（如社交媒体帖子），根据核心主题提炼中文标题，并以此为基础生成拼音或英文 slug。
- **跨领域文章**：以文章最核心的讨论主题确定单个 category，不宜拆入多个目录。
- **上下文截断**：若原文超出上下文窗口导致无法一次性完整萃取，优先保证"核心论点"与"方法论"的完整性，在文末标注"因篇幅限制，部分次要技术细节已省略"。截断场景下仍须按工作流生成并保存为 `.md` 文件。

### 完成前自检

生成 `.md` 文件后，逐项确认以下内容：

1. Frontmatter 格式正确：`---` 分隔符完整，`title`、`category`、`tags`、`date` 均已填写。
2. 文章已放入正确的 `articles/[category]/` 目录，slug 命名符合规范。
3. 全文通读无中文语病（主谓宾完整、无生硬直译）。
4. 原文中的所有代码、命令、配置片段均已原样保留，无遗漏、无翻译。
5. 结尾包含"一句话总结"（≤40 字）。
6. 已运行 `npm run gen` 刷新索引。

> 若用户明确表示只需在对话中浏览摘要而无需保存文件，则跳过步骤 4-8，直接在对话中输出完整 Markdown。

## 萃取与表达总则

### 信息密度与反幻觉

- **去噪存真**：去除口语化废话、重复性铺垫、过渡性寒暄，只保留有价值的观点、事实、数据、步骤与洞察。让读者无需阅读原文，也能掌握核心干货与关键洞察。
- **严禁编造**：严格基于原文提炼。若原文未涉及某板块，直接略过，不强行占位凑齐所有板块。

### 萃取覆盖范围

以下类别若原文有所涉及则必须覆盖；若确实没有则直接略过：

- **核心论点**：文章最想传达的 1-3 个中心思想
- **前置知识背景**：理解原文所需的基础知识（如特定工具、框架或理论前提）
- **关键概念与定义**：新名词、新方法论、专有缩写及语境解释
- **方法论 / 最佳实践**：可操作的步骤、流程、原则
- **技术细节与常见陷阱**：代码逻辑、架构决策、坑点、注意事项

### 术语、隐喻与专有名词

- **隐喻还原**：非技术词汇隐喻（如 Constitution、Silver bullet、Greenfield project）须转换为国内技术圈通行的标准表达（如"项目核心准则"、"银弹/万能药"、"从零新建项目"），不得生硬直译。无公认译法的隐喻保留原文并括号补充技术含义。
- **多义词消歧**：多义词（release、handle、consume、context 等）根据具体子领域的技术场景选择最准确的中文表达。
- **术语规范**：专业术语对齐国内主流技术社区及官方中文文档的通用译法，禁止自造词。缩写首次出现时须补充中文全称或解释（业界极通用缩写如 API、HTTP、JSON、SQL 除外）。
- **框架/工具/API 名称**：保留英文原样及官方大小写（如 Cursor、Replit、React、Supabase、Kubernetes）。
- **术语标注格式**：文章中首次出现的关键术语、方法论、框架别名，采用 **中文（English）** 格式标注一次，后续直接使用中文或英文原名。示例：依赖注入（Dependency Injection）、思维链（Chain-of-Thought）。
- **领域术语纠偏**：以下为常见误译对照，同类问题在其它子领域同样适用：
  - AI/LLM 语境：Agent → 智能体（非"代理"），Prompt → 提示词，Hallucination → 幻觉，Context window → 上下文窗口，Agentic engineering → 智能体工程，Vibe Coding → 氛围编程

### 代码与原文保留

- 所有原文中的**代码示例、命令行、配置文件片段、API 名称、文件路径、ASCII 目录树结构**必须原样保留，不做任何省略、翻译或改动。
- 对比性内容（如声明式 vs 命令式，有锁 vs 无锁并发），优先使用 Markdown 表格呈现。

### 输出格式

- 全程使用 Markdown 排版。
- H1 标题：原文有明确标题则沿用；否则提炼包含关键词的中文标题。
- 使用 H2/H3 标题划分知识板块（如"核心论点"、"关键概念"、"方法论与最佳实践"、"技术细节与陷阱"等），辅以列表、加粗、引用块（`>`）突出信息层级。
- 结尾必须附上 **"一句话总结"**（≤40 字）。若文章有明显受众区分（如"适合 Vibe Coding 新手"），可额外加一句 **"对你有何价值"**，点明适用场景。
